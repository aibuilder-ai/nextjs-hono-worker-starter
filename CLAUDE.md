# next-cloud — Cloudflare + Next.js Boilerplate

Next.js 16 app deployed to Cloudflare Workers via `@opennextjs/cloudflare`, with a standalone Hono API worker connected through a service binding.

## How the Service Binding Works

The Next.js worker calls `api-worker` directly — zero latency, same datacenter, no DNS, no cost.

```
wrangler.toml          [[services]] binding = "API_WORKER", service = "api-worker"
        ↓
env.d.ts               API_WORKER: Fetcher   ← auto-generated by cf-typegen
        ↓
src/app/api/items/route.ts
    const { env } = await getCloudflareContext({ async: true })
    env.API_WORKER.fetch(new Request("https://api-worker/items"))
```

The hostname in the Request URL is arbitrary — service bindings route by binding name, not DNS.

## Key Commands

> Always use **pnpm**. Never use npm or yarn in this project.

| Command | Description |
|---------|-------------|
| `pnpm dev:all` | Start all dev servers (Next.js + all workers) in parallel |
| `pnpm dev` | Next.js dev server only (local, no Cloudflare runtime) |
| `pnpm preview` | Build + run locally with full Cloudflare runtime |
| `pnpm deploy:all` | Deploy workers first, then the Next.js app |
| `pnpm deploy` | Deploy Next.js worker only (assumes workers already deployed) |
| `pnpm deploy:workers` | Deploy all workers in `workers/` only |
| `pnpm cf-typegen` | Regenerate `env.d.ts` from `wrangler.toml` |
| `pnpm build` | Next.js production build |
| `cd workers/api && pnpm dev` | Run Hono worker locally with wrangler dev |
| `cd workers/api && pnpm deploy` | Deploy Hono worker individually |

## Adding a New Worker

1. Create `workers/<name>/` with `src/index.ts`, `wrangler.toml` (set `name`), `package.json`
2. Add a `[[services]]` entry in the root `wrangler.toml`:
   ```toml
   [[services]]
   binding = "MY_WORKER"
   service = "<name>"
   ```
3. Run `npm run cf-typegen` — `MY_WORKER: Fetcher` will appear in `env.d.ts`
4. Call it from any Next.js route via `env.MY_WORKER.fetch(...)`
5. Add `cd workers/<name> && npm run deploy` to the `deploy:workers` script in root `package.json`

## Adding a Binding (KV, D1, R2, etc.)

1. Add the binding to `wrangler.toml` (e.g. `[[kv_namespaces]]`)
2. Run `npm run cf-typegen` to update `env.d.ts`
3. Access via `env.MY_BINDING` inside any Worker or Next.js edge route using `getCloudflareContext()`

## Rules for AI Agents

- **Deploy order**: always `npm run deploy:workers` before `npm run deploy` (or use `deploy:all`). Service bindings require the target worker to exist first.
- **Never edit `env.d.ts`** — it is fully generated by `wrangler types`. Regenerate it with `npm run cf-typegen` after any `wrangler.toml` change.
- **Workers in `workers/` are independent packages** — they have their own `node_modules`. Run `pnpm install` inside each worker directory separately.
- **`wrangler.toml` is the single source of truth** for all bindings and worker config. JSONC is not used.
- **Service binding URLs are fake hostnames** — `https://api-worker/path` routes via the binding name, not real DNS. Any hostname works.
- **`export const runtime = "edge"`** is required on Next.js route handlers that use `getCloudflareContext()`.
