# nextjs-hono-worker-starter

> Next.js + Hono on Cloudflare Workers вЂ” boilerplate with service bindings, typed API, and shadcn/ui.

A production-ready full-stack starter that runs **Next.js 16** on **Cloudflare Workers** via [`@opennextjs/cloudflare`](https://opennext.js.org/cloudflare), with a standalone **Hono** API worker connected through a zero-latency **service binding**. Types are shared between the worker and the Next.js app. UI is built with **shadcn/ui** and **Tailwind CSS**.

## How It Works

Next.js API routes call the Hono worker through a **Cloudflare service binding** вЂ” no HTTP round-trip, no DNS, no cost. The call is resolved within the same datacenter.

```
wrangler.toml  в†’  [[services]] binding = "API_WORKER", service = "api-worker"
env.d.ts       в†’  API_WORKER: Fetcher  (auto-generated)
route.ts       в†’  env.API_WORKER.fetch(new Request("https://api-worker/items"))
```

In local development (`pnpm dev:all`), the binding falls back to direct HTTP against the wrangler dev server on `localhost:8787`.

## Getting Started

**Prerequisites:** Node.js 18+, pnpm, a [Cloudflare account](https://dash.cloudflare.com/sign-up), Wrangler authenticated (`npx wrangler login`)

### 1. Install dependencies

```bash
pnpm install
cd workers/api && pnpm install && cd ../..
```

### 2. Start local development

```bash
pnpm dev:all
```

Starts Next.js on `localhost:3000` and the Hono worker on `localhost:8787` in parallel.

### 3. Deploy

```bash
pnpm deploy:all
```

Deploys the Hono worker first (required), then the Next.js worker. The deployed URL is printed by Wrangler.

## Adding a New Worker

1. Create `workers/<name>/` with `src/index.ts`, `src/types.ts`, `wrangler.toml`, `package.json`
2. Add to root `wrangler.toml`:
   ```toml
   [[services]]
   binding = "MY_WORKER"
   service = "<name>"
   ```
3. Run `pnpm cf-typegen` вЂ” `MY_WORKER: Fetcher` appears in `env.d.ts`
4. Call from any Next.js route via `env.MY_WORKER.fetch(...)`
5. Add `cd workers/<name> && pnpm run deploy` to the `deploy:workers` script

## Adding a Binding (KV, D1, R2, etc.)

1. Add to root `wrangler.toml` (e.g. `[[kv_namespaces]]`)
2. Run `pnpm cf-typegen` to update `env.d.ts`
3. Access via `env.MY_BINDING` in any route using `getCloudflareContext()`

## Sharing Types Between Workers and Next.js

Types live in `workers/<name>/src/types.ts` вЂ” no runtime imports, only `export type` declarations.

```typescript
// workers/api/src/types.ts
export type Item = { id: number; name: string; createdAt: string };

// src/app/api/items/route.ts
import type { Item } from "../../../../workers/api/src/types";
```

**Why not import from `index.ts`?** The worker's `index.ts` imports `hono`, which only exists in `workers/api/node_modules`. The Next.js bundler would try to follow that import and fail. `types.ts` has zero dependencies and is erased at build time.

---

## Gotchas

- **Deploy order matters** вЂ” `api-worker` must exist in your Cloudflare account before the Next.js worker can reference it via service binding. Always use `pnpm deploy:all`.
- **Never use `export const runtime = "edge"`** in Next.js route handlers. `@opennextjs/cloudflare` runs the entire app on Workers вЂ” adding this directive silently excludes the route from the server bundle.
- **`env.d.ts` is generated** вЂ” never edit it by hand. Run `pnpm cf-typegen` after any `wrangler.toml` binding change.
